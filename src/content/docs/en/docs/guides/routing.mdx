Open **Inference → Routes**. **Model routes** define how a client **`model`** (and optional **route group**) maps to **model + vendor + upstream model name**, and in which **priority** order to try them. Multiple routes in the same group enable automatic failover.

The list supports rich filters; the current filter set is reflected in the browser address bar so you can share a link to the same view.

<Aside type="tip" title="Prerequisite">
	Complete [Providers](/en/docs/guides/providers/) and [Models](/en/docs/guides/models/) first.
</Aside>

## What to do in Admin

<Steps>
1. Open **Routes**.
1. **Create a route**: pick model, vendor, upstream model name, **protocol** (must match the base URL you configured for that protocol on the provider), **route group** (often `default`), **priority** (lower tries first), **status** (only **active** routes serve production traffic).
1. **Billing**: set base multipliers for **charged to user** and **supplier cost** relative to the catalog list price. For vendor peak / off-peak rates, add a **Daily schedule** (time-of-day multipliers; timezone comes from [System config](/en/docs/guides/system-config/) business timezone).
1. After saving, use [Playground](/en/docs/features/playground/) for a single-route upstream check; use [Simulator](/en/docs/features/simulator/) to exercise the full Proxy path with a real `sk-` key.
</Steps>

## Client impact

Clients only send **`model`** and protocol-specific payloads — no hard-coded vendor details. Prefer changing routes when switching upstreams.

## Note

**Playground** probes **one** route upstream and does **not** mirror full multi-route failover inside the gateway; validate failover with real Proxy calls or **Simulator**.
