**Proxy** and **Admin** are separate **Worker** projects sharing one **D1** database (binding **`DB`**). You no longer hand-edit committed `wrangler.jsonc` files—**`npm run gen:wrangler`** generates gitignored wrangler configs from **Build variables** or **`cloudflare-worker/*.env`**.

<Aside type="note" title="Canonical details">
	Build watch paths, multi-instance forks, error **10181**, and full ops notes: [GitHub technical reference](/en/docs/github-reference/) → **deployment-cloudflare** and **cloudflare-worker/README**.
</Aside>

## Prerequisites

- Cloudflare account and **`wrangler login`** (or API token).
- Cloned [octafuse-gateway](https://github.com/OctaFuse/octafuse-gateway) with working `npm install` at repo root.
- For remote deploy: a **D1 database** (`npx wrangler d1 create`) and **`D1_DATABASE_ID`** in Build variables or a gitignored instance env—**do not commit production UUIDs to Git**.

## Configuration model

| Layer | Role |
|-------|------|
| `packages/*/wrangler.base.jsonc` | Committed templates (no production `database_id`) |
| `packages/proxy/wrangler.jsonc`, etc. | **Generated** (gitignored) by `gen:wrangler` |
| [`cloudflare-worker/example.env`](https://github.com/OctaFuse/octafuse-gateway/blob/main/cloudflare-worker/example.env) | **Dev demo** canonical config (committed) |
| `cloudflare-worker/<instance>.env` | **Production/private** (gitignored); or Dashboard **Build variables** only |

`gen-wrangler` reads **environment variables** (`process.env`) only—it does not care whether they come from the Dashboard or `dotenv -e`.

## Pick a path

| Path | When | Need `cloudflare-worker/*.env`? |
|------|------|----------------------------------|
| **A. Local dev** | Code on laptop, local D1 | ❌ No |
| **B. Dev demo** | Public test `test-api.octafuse.dev` | ✅ [`example.env`](https://github.com/OctaFuse/octafuse-gateway/blob/main/cloudflare-worker/example.env) |
| **C. Production** | Your own environment | ❌ Not in Git; use **Build variables** or gitignored env |

### Path A: Local Cloudflare development

See [Quick start · Option B](/en/docs/quick-start/): from repo root run `npm install`, `npm run db:migrate`, `npm run dev:proxy` (`:8787`), and `npm run dev:admin` (`:8789`). **No** `cloudflare-worker/*.env` required.

### Path B: Dev demo (CLI)

Uses repo [`cloudflare-worker/example.env`](https://github.com/OctaFuse/octafuse-gateway/blob/main/cloudflare-worker/example.env) (Worker names `*-dev`, D1 `octafuse-gateway-dev`).

<Steps>
1. **Create D1** (if needed): `npx wrangler d1 create octafuse-gateway-dev`; use `npx wrangler d1 list` and set **`D1_DATABASE_ID`** in `example.env`.
1. **Create Workers**: in Dashboard, create **`octafuse-gateway-proxy-dev`** and **`octafuse-gateway-admin-dev`** matching the env file.
1. **Set Secret**: `npx wrangler secret put ADMIN_PASSWORD --name octafuse-gateway-admin-dev`.
1. **Migrate and deploy** (repo root):
   ```bash
   npx dotenv -e ./cloudflare-worker/example.env -- npm run db:migrate:remote
   npx dotenv -e ./cloudflare-worker/example.env -- npm run deploy:proxy
   npx dotenv -e ./cloudflare-worker/example.env -- npm run deploy:admin
   ```
1. **Smoke test**: `GET https://test-api.octafuse.dev/health`, Admin login, one chat call with a test key.
</Steps>

Dev demo uses **CLI releases only**. When new SQL lands, run `db:migrate:remote` before `deploy:*`.

### Path C: Production (CLI or Git auto-deploy)

<Steps>
1. **Instance config**: copy `example.env` to `cloudflare-worker/<your-instance>.env` (**do not commit**), or set Dashboard **Build variables** only (same variable names).
1. **Fill env vars** (table below); **`D1_DATABASE_ID` lives in Dashboard or gitignored env only**.
1. **First bootstrap** (optional CLI, same generation as CI):
   ```bash
   npx dotenv -e ./cloudflare-worker/<instance>.env -- npm run db:migrate:remote
   npx dotenv -e ./cloudflare-worker/<instance>.env -- npm run deploy:proxy
   npx dotenv -e ./cloudflare-worker/<instance>.env -- npm run deploy:admin
   ```
1. **Connect to Git** (optional): connect **Proxy and Admin separately**; leave **Root directory empty** (monorepo root)—**do not** set it to `packages/proxy`.
1. **Build variables**: one set per Worker, same names/values as your instance env.
1. **Smoke test**: `GET /health`, Admin login, align downstream `GATEWAY_URL` / `GATEWAY_MASTER_URL` / `GATEWAY_MASTER_KEY` ([Integration](/en/docs/guides/integration/)).
</Steps>

## Environment variables (Build variables / local `.env`)

| Variable | Purpose |
|----------|---------|
| `PROXY_WORKER_NAME` / `ADMIN_WORKER_NAME` | **Must match Dashboard Worker names** |
| `D1_DATABASE_NAME` | D1 logical name |
| `D1_DATABASE_ID` | Required for remote deploy / migrate |
| `D1_MIGRATIONS_WORKER_NAME` | Optional; D1 migrations config name only—**no Worker to create** |
| `PROXY_CUSTOM_DOMAIN` / `ADMIN_CUSTOM_DOMAIN` | Optional; omit to bind hostnames in Dashboard |

## Workers Builds commands

**Leave Root directory empty** (monorepo root). `npm ci` → `postinstall` → `gen:wrangler` reads Build variables and writes `wrangler.jsonc`.

| Worker | Build command | Deploy command |
|--------|---------------|----------------|
| **Proxy** | `npm ci && npm run gen:wrangler` | `npm run deploy -w @octafuse/proxy` |
| **Admin** | `npm ci && npm run gen:wrangler && npm run build:cf -w @octafuse/admin` | `cd packages/admin && npx opennextjs-cloudflare deploy` |

<Aside type="caution" title="D1 migrations are not in CI">
	Git pipelines **do not** run D1 migrations. When new SQL ships, **`db:migrate:remote`** (with instance env) **first**, then `git push` or local `deploy:*`.
</Aside>

## Secrets and downstream

- **`ADMIN_PASSWORD`**: `npx wrangler secret put ADMIN_PASSWORD --name <ADMIN_WORKER_NAME>`—never commit or put in wrangler `vars`.
- Downstream portals: `GATEWAY_URL` (Proxy root), `GATEWAY_MASTER_URL` (Admin root), `GATEWAY_MASTER_KEY` (D1 `system_config.MASTER_KEY`).

## vs Docker

- **Edge + managed D1** → Cloudflare (this page).  
- **Strict VPC / BYO DB** → [Docker](/en/docs/deploy/docker/).
