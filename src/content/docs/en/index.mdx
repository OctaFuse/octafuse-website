Model access is getting fragmented: providers, compatible endpoints, coding plans, token plans, self-hosted models, and internal accounts all bring their own **Base URLs**, keys, quotas, pricing, and logs. Applications usually want something simpler: one stable entrypoint, controlled routing, clear budgets, and traceable request history.

Managed AI gateways usually revolve around a platform-owned model catalog, provider list, and control plane, so provider onboarding, routing policy, billing semantics, and extension points tend to stay inside that platform boundary. **OctaFuse** is designed as a self-hostable, modifiable gateway layer instead: connect official providers, third-party compatible endpoints, private model services, and multiple upstream keys while keeping routing, limits, circuit breaking, budgets, audit trails, and Admin APIs in your own system. Clients keep calling familiar OpenAI / Anthropic / Gemini-compatible APIs.

## Application Scenarios

<CardGrid stagger>
	<Card title="Individual developers" icon="add-document">
		Consolidate model subscriptions, coding plans, token plans, and backup upstreams behind one Gateway. Cursor, CLIs, scripts, and personal tools only need one Base URL and one user key.
	</Card>
	<Card title="AI apps and agents" icon="laptop">
		Build against stable model IDs and common protocols. Move model swaps, canary routes, backup providers, degradation, and default parameters into the gateway layer.
	</Card>
	<Card title="Platform and IT teams" icon="setting">
		Create separate users and API keys for departments, projects, members, or customers, with budget periods, status, and metadata outside application services.
	</Card>
	<Card title="API proxy businesses" icon="external">
		Connect multiple upstream providers and key pools, issue downstream customer keys, and manage each customer through request logs, cost lenses, and audit records.
	</Card>
	<Card title="Private and hybrid models" icon="setting">
		Put self-hosted compatible services, internal models, Ollama / vLLM paths, and public model providers behind one routing system with a consistent external surface.
	</Card>
	<Card title="Cost and reliability ops" icon="analytics">
		Observe usage and failures by model, Provider, user, and reliability view, then tune default routes, backups, budgets, and upstream key quotas.
	</Card>
</CardGrid>

## Product Capabilities

<CardGrid stagger>
	<Card title="Three protocol entrypoints" icon="external">
		Supports OpenAI Chat Completions, Anthropic Messages, Gemini `generateContent` / `streamGenerateContent`, plus `GET /v1/models` and public catalog discovery.
	</Card>
	<Card title="Models and route groups" icon="random">
		Clients can use stable model IDs or select routes with `baseId:group`. One model can attach multiple upstream routes by protocol, route group, priority, and weight.
	</Card>
	<Card title="Provider key scheduling" icon="approve-check">
		A Provider can hold multiple upstream keys. Scheduling combines route priority, key priority, headroom, circuit state, and sticky bindings before failover.
	</Card>
	<Card title="Limits, circuits, sticky" icon="warning">
		Upstream keys support **RPM / TPM / max concurrency**. 429, 401 / 403, 5xx, and network errors use separate cooldown policies; sticky bindings can improve prompt cache locality.
	</Card>
	<Card title="User keys and budgets" icon="padlock">
		Budgets and reset periods belong to Users; each user can own multiple active API keys. Proxy checks budget before forwarding and records charges after usage is parsed.
	</Card>
	<Card title="Logs, audit, analytics" icon="magnifier">
		Request logs capture protocol, model, route group, Provider, provider key, tokens, status, and costs. Audit logs track budget charges, resets, and user / key lifecycle events.
	</Card>
	<Card title="Playground and Simulator" icon="open-book">
		Playground tests a single Admin route. Simulator uses a real user key from the browser to verify auth, routing, billing, and logs.
	</Card>
	<Card title="Deployment and integration" icon="rocket">
		Run on Cloudflare Worker + Pages + D1, or Node / Docker with PostgreSQL or MySQL 8. External portals can provision users, keys, and budgets through **`/api/admin/*`**.
	</Card>
</CardGrid>
